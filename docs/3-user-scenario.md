## 1. 문서 정보

### 1.1 개요
본 문서는 my-todolist 서비스의 세 가지 사용자 페르소나(김직원, 이학생, 박운영)가 실제로 서비스를 이용하는 상황을 스토리텔링 형식으로 기술합니다. 각 페르소나의 고유한 행동 패턴, 동기, 목표, 그리고 감정을 반영하여 구체적인 사용 시나리오를 제시합니다.

### 1.2 목적
- 사용자 중심의 제품 설계 검증
- 각 페르소나별 주요 사용 패턴 파악
- UI/UX 개선 포인트 발굴
- 기능 우선순위 결정의 근거 제공
- 개발팀의 사용자 이해도 증진

### 1.3 기대 효과
- 실제 사용 맥락 기반의 기능 설계
- 사용자 니즈 충족도 향상
- 제품-시장 적합성(Product-Market Fit) 검증
- 출시 후 사용자 만족도 향상

## 2. 페르소나 요약

### 2.1 김직원 (30세, IT 과장, 부산)
- **배경**: 부산의 IT 회사에서 프로젝트 과장으로 근무
- **문제**: 일일 업무 태스크가 많아 할일 관리가 어려움, 메모 손실 경험
- **니즈**: 간단하게 할일을 메모하고 마감일 기준으로 우선순위 파악
- **사용 패턴**: 아침 업무 시작 시 할일 목록 확인, 낮 시간에 진행상황 업데이트, 저녁에 완료 체크

### 2.2 이학생 (25세, 대학교 4학년, 서울)
- **배경**: 서울 소재 대학교 마지막 학기 재학 중
- **문제**: 과제, 시험, 동아리 활동 등 다양한 할일 관리의 혼란
- **니즈**: 모든 할일을 한 곳에서 관리, 마감일 임박 상황을 시각적으로 확인
- **사용 패턴**: 스마트폰으로 학교 이동 중 빠르게 할일 추가, 집에서 PC로 상세 편집

### 2.3 박운영 (50세, 카페 사업가, 서울)
- **배경**: 서울 강남에서 소규모 카페를 운영 중
- **문제**: 재고 관리, 직원 급여 처리, 매월 세무 신고 등 반복적 할일 놓침
- **니즈**: 비용 최소화하면서 간편한 관리 도구 필요
- **사용 패턴**: 주 1회 정도 정기 회의 시 할일 추가, 월말에 집중해서 할일 확인

## 3. 페르소나별 주요 시나리오

## 3.1 김직원의 하루 - 업무 할일 관리

### 배경과 상황

**시간**: 2026년 2월 11일(수) 오전 8시 50분
**장소**: 부산 해운대구 IT회사 6층 사무실
**상황**: 새 프로젝트 첫 주, 팀장에게 이번 주 마감 업무 3가지를 방금 전달받음

김직원은 회사 책상에 도착해 커피를 마시며 업무를 시작하려 한다. 방금 팀장에게 이번 주 달성해야 할 업무 세 가지를 전달받았다:
1. 클라이언트 요구사항 정리 (금요일 마감)
2. 개발팀과 기술 스펙 검토 (목요일 마감)
3. 프로젝트 예산 검토안 작성 (수요일 마감)

김직원은 지난주 메모를 스마트폰에서 잃어버려 중요한 할일을 놓친 경험이 있었다. "이번엔 제대로 관리해야지. 어라, 동료 박준호가 좋은 할일 관리 앱을 쓰고 있다고 했는데... 그게 뭐였더라?"라고 생각하며 노트북을 켜고 구글을 검색한다.

### 시나리오 진행

#### 1단계: my-todolist 발견 및 회원가입 (08:55)

구글 검색 결과 중 "my-todolist - 간단한 개인 일정 관리"라는 배너를 발견한다. 사이트를 클릭하면 깔끔한 랜딩 페이지가 나타난다.

**김직원의 생각**: "오, 깔끔하네. 복잡한 기능 없이 정말 할일만 관리하는 거구나."

회원가입 버튼을 클릭한다. 회원가입 폼이 나타난다:
- 아이디 입력 필드: "kim_jikwon" 입력
- 비밀번호 입력 필드: "work2026!busan" 입력
- 비밀번호 확인: 동일하게 입력
- "가입하기" 버튼 클릭

**시스템 반응**: "회원가입이 완료되었습니다. 로그인 페이지로 이동합니다."

페이지가 로그인 화면으로 자동 이동된다. 방금 가입한 아이디와 비밀번호를 입력하여 로그인한다.

**김직원의 감정**: "와, 진짜 간단하네. 복잡한 인증 절차나 이메일 검증 없이 바로 쓸 수 있다니."

#### 2단계: 첫 할일 등록 (09:02)

로그인 후 할일 목록 페이지가 나타난다. 페이지 중앙에는 "할일이 없습니다"라는 메시지와 "새 할일 추가" 버튼이 있다.

**김직원의 생각**: "자, 이제 팀장에게서 받은 세 가지 일을 입력해야지."

"새 할일 추가" 버튼을 클릭하면 할일 입력 폼이 나타난다:

**첫 번째 할일 입력:**
- 제목: "프로젝트 예산 검토안 작성"
- 설명: "클라이언트 예산선 내에서 현실적인 예산안 3개 버전 작성 필요"
- 마감일: 2026-02-12 (내일)
- "추가하기" 버튼 클릭

**시스템 반응**: "할일이 추가되었습니다."

새로운 할일이 목록에 나타난다. 제목과 마감일이 표시되고, 마감일이 빨간 글씨로 강조되어 있다 (내일이 마감이므로).

**두 번째 할일 입력:**
- 제목: "개발팀과 기술 스펙 검토"
- 설명: "API 설계, 데이터베이스 스키마, 보안 요구사항 검토"
- 마감일: 2026-02-13 (모레)
- "추가하기" 클릭

**세 번째 할일 입력:**
- 제목: "클라이언트 요구사항 정리"
- 설명: "회의 녹음본 검토 후 주요 기능 10개 정리, PPT 작성"
- 마감일: 2026-02-14 (금요일)
- "추가하기" 클릭

이제 할일 목록에 세 개의 할일이 마감일 순서대로 정렬되어 있다. 가장 마감이 임박한 "프로젝트 예산 검토안 작성"이 맨 위에 있다.

**김직원의 감정**: "좋다. 이제 뭐를 먼저 할지 명확하네. 마감일순으로 정렬되어 있어서 우선순위를 한눈에 파악할 수 있다."

#### 3단계: 업무 진행 중 할일 수정 (11:30)

오전 10시부터 개발팀과 미팅을 했다. 예상했던 것보다 기술 스펙 검토가 더 복잡해 보인다. 김직원은 해당 할일의 기한을 연장해야 한다고 생각한다.

"개발팀과 기술 스펙 검토" 할일 옆의 수정 버튼을 클릭한다. 수정 폼이 나타난다:
- 제목: 그대로 유지
- 설명: "API 설계, 데이터베이스 스키마, 보안 요구사항, 성능 최적화 전략 검토"로 수정
- 마감일: 2026-02-13 에서 2026-02-14로 변경 (금요일로 연장)
- "수정하기" 클릭

**시스템 반응**: "할일이 수정되었습니다."

목록이 자동으로 재정렬된다. 이제 "클라이언트 요구사항 정리"와 "개발팀과 기술 스펙 검토"가 같은 금요일 마감이지만, "클라이언트 요구사항 정리"가 먼저 나타난다.

**김직원의 감정**: "오, 자동으로 마감일순으로 다시 정렬되네. 좋다."

#### 4단계: 점심시간 - 모바일로 빠른 확인 (12:30)

김직원은 점심을 먹으며 스마트폰으로 my-todolist에 접속한다. 로그인은 자동으로 유지되고 있다 (localStorage에 저장된 JWT 토큰).

모바일 화면도 깔끔하게 표시된다. 세 개의 할일이 모두 보이고, 마감일이 가장 임박한 "프로젝트 예산 검토안 작성"이 맨 위에 있다.

**김직원의 생각**: "좋아, 아직 할일은 진행되지 않았군. 오후 2시부터 예산안 작성에 집중해야지."

#### 5단계: 오후 업무 - 첫 할일 완료 (17:45)

오후 내내 집중해서 일한 김직원은 드디어 첫 번째 할일인 "프로젝트 예산 검토안 작성"을 완료했다.

my-todolist에 접속하여 해당 할일 옆의 체크박스를 클릭한다.

**시스템 반응**: 할일이 완료 상태로 변경된다. 할일 텍스트에 취소선이 그어지고 회색으로 표시된다. 체크 아이콘이 나타난다.

**김직원의 감정**: "오! 완료했다! 뭔가 신기한데... 체크박스를 누르니까 자동으로 이렇게 표시되네. 성취감이 느껴진다."

완료된 할일은 목록의 맨 아래로 이동한다. 이제 목록의 맨 위에는 "개발팀과 기술 스펙 검토"가 표시된다.

#### 6단계: 퇴근 전 하루 정리 (18:00)

퇴근하기 전에 김직원은 my-todolist를 다시 열어 하루를 정리한다.

현재 상황:
- 완료됨: "프로젝트 예산 검토안 작성" (1개)
- 진행 중: "개발팀과 기술 스펙 검토" (금요일 마감), "클라이언트 요구사항 정리" (금요일 마감)

**김직원의 생각**: "내일은 기술 스펙 검토를 완료해야 하고, 금요일까지 클라이언트 요구사항 정리를 끝내면 돼. 계획이 명확하니까 좋아."

## 3.2 이학생의 학업 관리 - 대학교 마지막 학기

### 배경과 상황

**시간**: 2026년 2월 11일(수) 오전 10시 15분
**장소**: 서울 강남구 대학교 도서관
**상황**: 4학년 2학기, 졸업 전 마지막 학기, 과제 5개 + 시험 3개 + 졸작 프로젝트 진행 중

이학생(25세)은 도서관에서 노트북을 켜고 강의 자료를 정리하고 있다. 이번 학기는 정말 바쁘다:
- 데이터베이스 과목: 팀 프로젝트 + 시험 (3월 5일)
- 소프트웨어 엔지니어링: 과제 2개 + 최종 시험 (2월 28일)
- 졸업 프로젝트: 팀원 3명과 함께 진행 중 (4월 30일 최종 발표)
- 동아리: 정기 회의 (매주 목요일)
- 자격증 준비: 정보처리기사 필기 (3월 22일)

스마트폰 메모장에는 할일들이 뒤죽박죽 섞여 있다. 어떤 것을 먼저 해야 할지 혼란스럽다.

**이학생의 생각**: "아, 이게 뭐지... 메모들이 너무 많아서 뭐가 중요한지 모르겠어. 어제 인스타그램에서 봤는데 할일 관리 앱이 있다고 했는데... my-todolist라는 게 있었나?"

### 시나리오 진행

#### 1단계: 모바일에서 회원가입 (10:20)

도서관에서 스마트폰으로 my-todolist를 검색하여 사이트에 접속한다. 모바일 화면이 깔끔하게 표시된다.

"회원가입" 버튼을 탭한다. 모바일 친화적인 회원가입 폼이 나타난다:
- 아이디: "leestudent25" 입력
- 비밀번호: "study2026!seoul" 입력
- "가입하기" 버튼 탭

**시스템 반응**: "회원가입이 완료되었습니다."

자동으로 로그인되어 할일 목록 페이지로 이동한다.

**이학생의 감정**: "오, 모바일에서도 이렇게 잘 작동하네! 터치하기에도 편하다."

#### 2단계: 모바일에서 빠른 할일 추가 (10:25)

도서관에서 손가락으로 빠르게 할일을 입력해 나간다. "새 할일 추가" 버튼을 반복해서 탭한다.

**추가된 할일들:**
1. 제목: "데이터베이스 팀 프로젝트 1차 결과물 제출"
   마감일: 2026-02-20

2. 제목: "소프트웨어 엔지니어링 과제 1 (요구분석서)"
   마감일: 2026-02-16

3. 제목: "정보처리기사 필기 시험"
   마감일: 2026-03-22
   설명: "암기 부분 정리, 기출 문제 5회 이상 풀기"

4. 제목: "졸업 프로젝트 API 설계"
   마감일: 2026-02-28

5. 제목: "동아리 정기 회의 발표 준비"
   마감일: 2026-02-13 (이번 목요일)

6. 제목: "소프트웨어 엔지니어링 최종 시험"
   마감일: 2026-02-28

7. 제목: "데이터베이스 최종 시험"
   마감일: 2026-03-05

**이학생의 감정**: "와, 모든 할일을 한 곳에 정리하니까 한결 마음이 편하다. 뭐가 중요한지 이제 명확하네."

목록은 자동으로 마감일순으로 정렬된다:
1. 동아리 정기 회의 발표 준비 (2월 13일) ← 가장 임박
2. 소프트웨어 엔지니어링 과제 1 (2월 16일)
3. 데이터베이스 팀 프로젝트 1차 (2월 20일)
4. 소프트웨어 엔지니어링 최종 시험 (2월 28일)
5. 졸업 프로젝트 API 설계 (2월 28일)
6. 정보처리기사 필기 시험 (3월 22일)
7. 데이터베이스 최종 시험 (3월 05일)

#### 3단계: 도서관에서 PC로 상세 편집 (11:00)

5. **목록에 즉시 표시**: 새 할일이 추가되고 마감일순으로 재정렬

## 4.3 마감일 관리 및 지연 알림

### 시나리오: 마감일이 지난 할일의 시각화

**상황**: 2026년 2월 12일, 이학생이 my-todolist 확인

#### 마감일 지난 할일
"동아리 정기 회의 발표 준비"는 원래 2월 13일 마감이었는데, 아직 완료하지 않았다 (현재 날짜: 2월 14일).

**시스템 표시**:
- 배경색: 밝은 빨간색 (#ffebee)
- 텍스트: 검은색 (기본)
- 아이콘: 경고 표시 (⚠️)
- 시각적 강조: 목록에서 가장 눈에 띄는 위치

**사용자 인식**: "아, 이 할일은 마감이 지났네! 지금 당장 처리해야겠다!"

#### 완료 처리 시 변화
이학생이 할일을 완료하기로 결정하고 체크박스를 클릭한다.

**시스템 반응**:
- 빨간색 배경 제거
- 텍스트에 취소선 추가
- 텍스트가 회색으로 변함
- 경고 아이콘 제거, 체크 아이콘 추가

**사용자 감정**: "좋다, 지연된 할일도 완료했다!"

## 4.4 모바일 사용 시나리오

### 시나리오: 이학생의 모바일 중심 사용

**상황**: 2026년 2월 16일, 이학생은 교실 이동 중

#### 장면 1: 강의실에서 할일 추가 (10분)
강의가 끝나고 다음 강의까지 10분 시간이 남았다. 이학생은 스마트폰에서 my-todolist를 연다.

"새 할일 추가" 버튼을 탭한다. 모바일 최적화된 입력 폼이 나타난다:
- 제목 입력: 탭하면 온스크린 키보드 나타남
- 마감일 선택: 날짜 피커 나타남
- "추가하기" 버튼: 큼 (터치하기 쉬움)

**추가된 할일**: "인공지능 과제 읽기 및 정리"
마감일: 2026-02-23

버튼을 탭하면 1초 내에 할일이 추가되고 목록이 업데이트된다.

**이학생의 생각**: "좋다, 이렇게 빠르게 추가할 수 있으니 언제든 기억날 때 입력할 수 있겠다."

#### 장면 2: 도서관에서 할일 확인 (2분)
도서관에 들어가며 스마트폰을 꺼내 today's priority를 확인한다. 마감일이 가장 임박한 할일들이 위에 표시되어 있다.

화면을 위아래로 스크롤하며 할일 목록을 본다. 모바일 화면에서도 모든 할일이 명확하게 표시된다.

#### 장면 3: 과제 완료 후 체크 (1분)
도서관에서 과제를 완료하고 스마트폰을 꺼낸다. 해당 할일의 체크박스를 탭한다.

**시각적 변화**: 즉시 취소선과 회색 텍스트가 적용된다. 새로고침이 필요 없다.

**모바일 사용의 핵심 가치**:
- 언제 어디서나 빠르게 할일 추가 가능
- 장시간 노트북 사용하지 않아도 할일 추적 가능
- 터치 친화적 UI로 빠른 상호작용

## 5. 엣지 케이스 시나리오

## 5.1 에러 상황 및 대응

### 시나리오 1: 회원가입 중 중복 아이디

**사용자**: 새로운 사용자
**상황**: my-todolist에 가입하려고 함

#### 입력 내용
```
아이디: kim_jikwon (이미 존재하는 아이디)
비밀번호: password123
```

#### 시스템 반응
"가입하기" 버튼 클릭 시 409 Conflict 에러:
```
에러 메시지: "이미 사용 중인 아이디입니다. 다른 아이디를 입력해주세요."
```

#### 사용자 경험
- 에러 메시지가 빨간색으로 강조 표시됨
- 아이디 입력 필드가 빨간 테두리로 표시됨
- 입력한 비밀번호는 지워짐 (보안상 이유)
- "다른 아이디 시도" 방법 제시

**해결 방법**:
아이디를 "kim_jikwon_2" 또는 "kim_jikwon30" 등으로 변경하여 재시도

---

### 시나리오 2: 로그인 실패

**사용자**: 가입했으나 비밀번호를 잊어버린 사용자
**상황**: my-todolist 로그인 시도

#### 입력 내용
```
아이디: kim_jikwon
비밀번호: wrong_password
```

#### 시스템 반응
"로그인" 버튼 클릭 시 401 Unauthorized 에러:
```
에러 메시지: "아이디 또는 비밀번호가 일치하지 않습니다."
```

#### 사용자 경험
- 에러 메시지가 양쪽 필드 위에 붉은색으로 표시
- 보안 상 정확히 어느 필드가 잘못되었는지 명시하지 않음 (올바른 보안 관행)
- 로그인 버튼 아래에 "비밀번호 재설정" 링크 제시

**해결 방법**:
"비밀번호 재설정" (향후 기능) 또는 올바른 비밀번호로 재시도

---

### 시나리오 3: 할일 추가 시 필수값 누락

**사용자**: 김직원
**상황**: 할일 추가 폼에서 제목 없이 제출 시도

#### 시스템 반응
"추가하기" 버튼 클릭 시 400 Bad Request:
```

```

#### 사용자 경험
- 제목 입력 필드가 빨간 테두리로 강조
- 에러 메시지가 필드 아래에 표시
- 다른 필드는 유지됨 (재입력 불필요)

**해결 방법**:
제목을 입력한 후 다시 "추가하기" 클릭

마감일: "2026/02/15" (잘못된 형식, YYYY-MM-DD 형식 필수)

#### 시스템 반응
"추가하기" 클릭 시 400 Bad Request:
```
에러 메시지: "유효한 마감일을 입력해주세요. (형식: YYYY-MM-DD)"
```

#### 사용자 경험
- 마감일 필드가 빨간 테두리로 강조
- 올바른 형식 안내 메시지 표시
- 다른 필드는 유지됨

**해결 방법**:
날짜 피커 사용 (권장) 또는 올바른 형식으로 재입력

### 8.1 김직원 (직장인)

**주요 니즈**:
- 빠른 할일 추가 (아침 업무 시작 시)
- 마감일 기반 우선순위 파악
- 진행상황 추적 및 성취감

**사용 패턴**:
- 주로 데스크톱 (회사 노트북)
- 하루 3~4회 접속 (아침, 점심, 오후, 저녁)
- 할일 완료 시 즉시 체크

**설계 고려사항**:
- 빠른 로딩 속도 필수
- 마감일순 정렬이 가장 중요
- 간단한 UI (복잡한 기능 불필요)

### 8.2 이학생 (대학생)

**주요 니즈**:
- 다양한 할일 종류 관리 (과제, 시험, 프로젝트, 동아리)
- 마감일 임박 상황 시각화
- 어디서나 빠른 추가 (모바일)

**사용 패턴**:
- 모바일과 데스크톱 동시 사용
- 모바일로는 빠른 추가 및 확인
- PC에서는 상세 편집 및 관리
- 하루 5~6회 접속

**설계 고려사항**:
- 완벽한 반응형 디자인 필수
- 모바일 터치 인터페이스 최적화
- 마감일 지연 알림이 중요
- 설명 필드 충분한 크기

### 8.3 박운영 (자영업자)

**주요 니즈**:
- 정기적인 반복 할일 관리
- 팀원과의 할일 공유
- 월간 정리 및 통계

**사용 패턴**:
- 주 1회 정기 미팅에서 할일 입력
- 할일 추가는 적지만 중요도 높음
- 장기 마감일 (몇 주~몇 달)
- 주간 1회 진행상황 확인

**설계 고려사항**:
- 간단함이 최우선 (기술 이해도 낮음)
- 직관적 UI
- 오류 최소화
- 한국어 메시지 매우 중요

## 9. 사용자 경험 개선 포인트

### 9.1 발견된 기회

| 기회 | 페르소나 | 우선순위 | 설명 |
|------|---------|---------|------|
| 알림 기능 | 이학생 | 높음 | 마감일 1일 전 알림 기능 필요 |

| 암다크 모드 | 김직원 | 낮음 | 야간 모드 (향후) |
| 협업 기능 | 박운영 | 낮음 | 팀원과 할일 공유 (향후) |

### 9.2 사용자 만족도 예측

**박운영**: 88% (간단함, 한국어 인터페이스)

## 결론

my-todolist는 세 가지 서로 다른 사용자 페르소나의 실제 니즈를 충족시키는 설계가 되어 있습니다:

1. **김직원**: 업무 중심의 간단함과 빠른 응답 속도 제공
2. **이학생**: 모바일 최적화와 마감일 관리 강화
3. **박운영**: 직관성과 한국어 UI로 접근성 확보

이 세 페르소나의 사용 시나리오를 통해 MVP 단계에서 반영해야 할 핵심 기능들을 검증하고, 향후 기능 개선 방향을 제시할 수 있습니다.
