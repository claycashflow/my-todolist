## 1. 문서 정보

### 1.1 개요
이 PRD는 my-todolist 프로젝트의 MVP(Minimum Viable Product) 범위와 기술 요구사항을 정의합니다. 도메인 정의서(1-domain-definition.md)의 비즈니스 규칙과 시나리오를 구현 가능한 수준으로 구체화했습니다.

### 1.2 주요 일정
- **개발 기간**: 3일 (화~목)
- **목표 출시**: 금요일 오후
- **MVP 범위**: 회원가입, 로그인, 할일 CRUD + 완료 처리

---

## 2. 제품 개요

### 2.1 제품 비전
개인 일정 관리를 위한 인증 기반 웹 애플리케이션으로, 사용자가 할 일을 작성하고 마감일을 관리하며 진행 상황을 추적할 수 있는 간단하고 직관적인 도구 제공

### 2.2 문제 정의
현대인들은 많은 할 일들을 관리해야 하지만:
- 종이나 부자연스러운 관리 도구로 인한 생산성 저하
- 마감일을 놓쳐 업무 지연 발생
- 여러 기기에서 동기화되지 않는 데이터 관리의 불편함

my-todolist는 웹 기반으로 언제 어디서나 개인 할 일을 효율적으로 관리할 수 있는 해결책 제공

### 2.3 성공 정의
**MVP 출시 성공 기준:**
- 회원가입 및 로그인 기능 정상 작동
- 할일 CRUD 기능 완벽 구현
- 마감일 지연 알림 정상 표시
- 반응형 UI로 모바일/데스크톱 모두 지원
- 신규 사용자가 별도 튜토리얼 없이 직관적 사용 가능

## 4. 제품 범위

### 4.1 MVP 포함 기능 (필수)

#### 4.1.1 사용자 인증
- **회원가입 (`/auth/register`)**
  - username 입력 (4~20자, 영문/숫자만 허용)
  - password 입력 (8자 이상, bcrypt로 해시화되어 저장)
  - 중복 username 검증 (409 Conflict 반환)
  - 성공 시 로그인 페이지로 리다이렉트

- **로그인 (`/auth/login`)**
  - username 및 password 입력
  - password 해시 비교를 통한 검증
  - 성공 시 JWT 토큰 발급 (유효기간 24시간)
  - 실패 시 401 Unauthorized + 오류 메시지
  - 토큰을 localStorage에 저장하여 자동 인증 유지

#### 4.1.2 할일 관리
- **할일 목록 조회 (`GET /todos`)**
  - 로그인한 사용자의 모든 할일 조회
  - 각 할일에 `isOverdue` 필드 포함 (마감일 < 오늘 && done=false)
  - 마감일 순서로 정렬 (가까운 순서)

- **할일 상세 조회 (`GET /todos/:id`)**
  - 특정 할일의 전체 정보 조회
  - 소유권 검증 (본인 할일만 접근 가능)

- **할일 추가 (`POST /todos`)**
  - 필수 입력: title (1~100자), dueDate (YYYY-MM-DD)
  - 선택 입력: description (최대 1000자)
  - done 기본값: false
  - 입력값 검증 후 201 Created 반환

- **할일 수정 (`PUT /todos/:id`)**
  - title, description, dueDate, done 필드 수정 가능
  - 부분 수정 지원 (변경할 필드만 전송 가능)
  - 소유권 검증 후 200 OK 반환
  - updatedAt 자동 갱신

- **할일 완료 처리 (`PUT /todos/:id` 또는 `PATCH /todos/:id`)**
  - done 필드를 토글 또는 직접 설정
  - 완료된 할일의 isOverdue 표시 제거

- **할일 삭제 (`DELETE /todos/:id`)**
  - 소유권 검증 후 영구 삭제
  - 204 No Content 반환
  - 삭제 전 "정말 삭제하시겠습니까?" 확인 UI

#### 4.1.3 UI 및 UX
- **반응형 레이아웃**
  - 모바일 (< 576px), 태블릿 (576px ~ 992px), 데스크톱 (≥ 992px)
  - Bootstrap 또는 CSS Grid/Flexbox 활용

- **마감일 지연 알림**
  - isOverdue=true인 할일: 빨간색 배경 (#ffebee) + 경고 아이콘
  - 완료된 할일: 취소선 + 회색 텍스트 + 체크 아이콘
  - 일반 할일: 기본 스타일

- **사용자 친화적 메시지**
  - 모든 에러 메시지는 한국어로 제공
  - 성공/실패 피드백 (토스트 또는 알림)

### 4.2 MVP 미포함 기능 (Phase 2 이후)

- 할일 카테고리/라벨 분류
- 반복 일정 (매일, 매주, 매달)
- 알림 및 이메일 푸시 알림
- 협업 기능 (할일 공유, 댓글)
- 할일 검색 및 필터링
- 다크 모드
- 사용자 프로필 수정
- 계정 비활성화/탈퇴
- 쿠키/세션 기반 인증 대체 (현재는 JWT만)

## 6. 인수기준 (Acceptance Criteria)

### 6.1 기능별 상세 인수기준

#### 6.1.1 회원가입 기능 (AC-001)

**기능 설명**: 신규 사용자가 계정을 생성할 수 있다

**인수기준**:
- [ ] username 입력 필드가 표시되며, 4~20자 제한이 적용된다
- [ ] password 입력 필드가 표시되며, 8자 이상 제한이 적용된다
- [ ] username이 4자 미만 또는 20자 초과 시 "아이디는 4~20자의 영문과 숫자로 입력해주세요" 메시지 표시
- [ ] password가 8자 미만 시 "비밀번호는 8자 이상 입력해주세요" 메시지 표시
- [ ] 영문/숫자 외 특수문자 입력 시 username 검증 실패
- [ ] 이미 존재하는 username 입력 시 409 Conflict + "이미 사용 중인 아이디입니다" 메시지
- [ ] 정상 가입 시 201 Created 응답 + DB에 사용자 레코드 생성
- [ ] password는 bcrypt 해시로 저장 (평문 저장 금지)
- [ ] 가입 성공 시 로그인 페이지로 자동 리다이렉트

**테스트 시나리오**:
```
Given: 회원가입 페이지 접속
When: username "testuser123", password "password123" 입력 후 제출
Then: 201 응답, DB에 사용자 생성, 로그인 페이지로 이동

Given: 회원가입 페이지 접속
When: 이미 존재하는 username "testuser123" 재입력
Then: 409 응답, "이미 사용 중인 아이디입니다" 에러 메시지 표시

Given: 회원가입 페이지 접속
When: username "ab" (2자) 입력
Then: "아이디는 4~20자의 영문과 숫자로 입력해주세요" 메시지 표시
```

#### 6.1.2 로그인 기능 (AC-002)

**기능 설명**: 기존 사용자가 인증하여 서비스에 접근할 수 있다

**인수기준**:
- [ ] username과 password 입력 필드가 표시된다
- [ ] 올바른 username + password 입력 시 JWT 토큰 발급
- [ ] 토큰은 localStorage에 저장되어 자동 인증 유지
- [ ] 잘못된 username 또는 password 입력 시 401 Unauthorized
- [ ] 401 에러 시 "아이디 또는 비밀번호가 일치하지 않습니다" 메시지 표시
- [ ] 로그인 성공 시 할일 목록 페이지로 자동 이동
- [ ] JWT 토큰 유효기간: 24시간
- [ ] 만료된 토큰 사용 시 자동 로그아웃 + 로그인 페이지로 이동

**테스트 시나리오**:
```
Given: 가입된 사용자 (username: "testuser", password: "password123")
When: 로그인 페이지에서 올바른 정보 입력

Given: 로그인 페이지 접속
When: 잘못된 password 입력
Then: 401 응답, "아이디 또는 비밀번호가 일치하지 않습니다" 메시지

Given: 24시간 경과한 토큰 보유
When: 인증이 필요한 페이지 접근
Then: 401 응답, 자동 로그아웃, 로그인 페이지로 리다이렉트
```

#### 6.1.3 할일 추가 기능 (AC-003)

- [ ] description 입력 필드 (선택, 최대 1000자)

- [ ] dueDate가 유효하지 않은 형식 시 400 에러 + "유효한 마감일을 입력해주세요"

```
When: title 빈 문자열로 제출
Then: 400 응답, "제목은 1~100자로 입력해주세요" 메시지

Given: 미인증 사용자
```

#### 6.1.4 할일 목록 조회 기능 (AC-004)

```
Then: "할일이 없습니다" 메시지 표시
```

- [ ] 완료 처리 시 즉시 스타일 변경 (빨간색 배경 제거)

**테스트 시나리오**:

```
When: 현재 날짜가 "2026-02-11"
Then: isOverdue=false, 일반 스타일

Given: 마감일이 "2026-02-09"인 완료된 할일 (done=true)
When: 현재 날짜가 "2026-02-10" 이후
Then: isOverdue=true이지만 빨간색 스타일 적용 안 함, 취소선만 표시
```

| password | 최소 8자 이상, bcrypt 해시화 저장 | "비밀번호는 8자 이상 입력해주세요" |

#### 6.2.2 할일(Todo) 검증

| 필드 | 규칙 | 에러 메시지 |

| dueDate | YYYY-MM-DD 형식, 필수 입력 | "유효한 마감일을 입력해주세요" |
| done | Boolean (기본값: false) | - |

### 6.3 권한 및 소유권 검증

- **BR-003**: 할일 생성, 조회, 수정, 삭제는 인증된 사용자만 가능
  - 미인증 시: 401 Unauthorized + "로그인이 필요합니다"
  - 구현: authMiddleware (모든 /todos/* 경로에 적용)

- **BR-004**: 사용자는 자신이 생성한 할일만 조회 및 수정할 수 있다
  - 권한 없음 시: 403 Forbidden + "권한이 없습니다"
  - 구현: ownershipMiddleware (GET, PUT, DELETE /todos/:id에 적용)

### 6.4 비즈니스 로직

#### 6.4.1 마감일 지연 계산 (BR-006)
```
isOverdue = (dueDate < today) && (done === false)
```
- 시간 제외 날짜만 비교 (00:00:00 기준)
- 마감일이 오늘인 경우: isOverdue = false
- 완료된 할일은 isOverdue 무시

#### 6.4.2 토큰 관리 (BR-002)
- 로그인 성공 시 JWT 토큰 발급
- 토큰 유효기간: 24시간
- 토큰은 요청 시 Authorization 헤더로 전달 (Bearer <token>)
- 만료된 토큰 사용 시: 401 Unauthorized

```sql
);
```

```sql
CREATE TABLE todos (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  description VARCHAR(1000),
  due_date DATE NOT NULL,
  done BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

);

CREATE INDEX idx_todos_user_id ON todos(user_id);
```

## 11. 리스크 및 제약사항

### 11.1 기술적 리스크

| 리스크 | 영향도 | 확률 | 대응 방안 |
|--------|--------|------|---------|
| PostgreSQL 연결 문제 | 높음 | 중 | 사전 로컬 테스트 후 배포 |
| JWT 토큰 만료 처리 실패 | 중간 | 중 | 프론트엔드에서 자동 새로고침 로직 구현 |
| React 렌더링 성능 저하 | 중간 | 낮음 | 컴포넌트 최적화 (React.memo, useMemo) |
| 네트워크 요청 타이밍 이슈 | 중간 | 중 | 요청 중복 방지 로직, 로딩 상태 관리 |

### 11.2 프로젝트 제약사항

- **개발 기간**: 3일 (Day 1~3 개발, Day 4 배포)
- **팀 규모**: 소규모 팀 (풀스택 개발자 1~2명 가정)
- **오버엔지니어링 금지**: 최소 필수 기능만 구현
- **복잡한 기능 제외**:
  - 데이터 마이그레이션 도구
  - 고급 데이터 분석
  - 협업 기능
  - 외부 서비스 연동
- **배포 환경**: Vercel 예정 (테스트 환경은 로컬)

### 11.3 비즈니스 제약사항

- **사용자 확보**: 초기에는 친구/지인 기반으로 테스트
- **마케팅**: MVP 출시 후 심화 (이번 주는 개발에 집중)
- **유료화**: MVP 단계에서는 제공 안 함
- **커뮤니티**: 초기에는 지원하지 않음
